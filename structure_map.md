# Карта проекта Cardio_marathon

## Общая архитектура

Проект представляет собой многокомпонентное веб-приложение с бэкендом на FastAPI, фронтендом на React/Vite, Telegram-ботом и инфраструктурой, оркестрируемой через Docker. Архитектура разделена на микросервисы, объединённые через `docker-compose.yml`.

---

## Корень проекта

- **`.github/`** — содержит конфигурации GitHub Actions для CI/CD (автоматические тесты, сборка образов, деплой).
- **`.gitignore`** — определяет файлы и директории, исключённые из контроля версий (например, `.venv/`, `__pycache__`, `.env`).
- **`.idea/`** — настройки IDE PyCharm (локальные, не требуют коммита, но могут быть полезны для командной разработки).
- **`.venv/`** — виртуальное окружение Python с установленными зависимостями (не коммитится, создаётся локально).
- **`docker-compose.yml`** — основной файл оркестрации Docker, объединяет сервисы: backend, frontend, bot, nginx.
- **`backend/`** — бэкенд-сервис на FastAPI, обрабатывает API-запросы, логику игры, OCR и взаимодействие с БД.
- **`frontend/`** — фронтенд-приложение на React с Vite, отображает интерфейс игры.
- **`bot/`** — Telegram-бот для уведомлений, управления и взаимодействия с игроками.
- **`nginx/`** — конфигурация веб-сервера Nginx для проксирования запросов и раздачи статики.

---

## backend/

- **`main.py`** — **точка входа** FastAPI-приложения. Запускает сервер, подключает роуты и middleware.
- **`config.py`** — конфигурационный файл: хранит настройки приложения (режим разработки, секретные ключи, пути).
- **`database.py`** — настройка подключения к базе данных через SQLAlchemy, инициализация сессий и engine.
- **`models.py`** — ORM-модели SQLAlchemy, описывают структуру таблиц в БД (игроки, прогресс, предметы и т.д.).
- **`schemas.py`** — Pydantic-схемы для валидации входных и выходных данных API.
- **`ocr_service.py`** — **модуль обработки данных**: реализует OCR для распознавания скриншотов активности (например, из приложения кардиотрекера).
- **`mechanics.py`** — **модуль бизнес-логики**: содержит игровые механики (начисление очков, проверка заданий, уровни).
- **`boss_factory.py`** — генерация и управление "боссами" в игре (динамические цели, баланс).
- **`shop_config.py`** — конфигурация виртуального магазина (цены, товары, эффекты).
- **`requirements.txt`** — зависимости Python для бэкенда.
- **`Dockerfile`** — инструкции для сборки Docker-образа бэкенда.

---

## frontend/

- **`src/`** — исходный код React-приложения: компоненты, хуки, стили, логика UI.
- **`index.html`** — **точка входа** фронтенда, подключает React и Vite.
- **`package.json`** — зависимости и npm-скрипты (запуск, сборка, тесты).
- **`vite.config.js`** — конфигурация сборщика Vite (прокси, плагины, режимы).
- **`nginx.conf`** — конфигурация Nginx для раздачи статики фронтенда.
- **`Dockerfile`** — образ Docker для фронтенда (сборка и экспорт статики).

---

## bot/

- **`bot.py`** — **точка входа** Telegram-бота. Обрабатывает команды, отправляет уведомления, интегрируется с бэкендом.
- **`requirements.txt`** — зависимости Python для бота (aiogram, requests и т.д.).
- **`Dockerfile`** — образ Docker для запуска бота.

---

## nginx/

- **`default.conf`** — основной конфиг Nginx: определяет проксирование `/api/*` на бэкенд, `/bot/*` на Telegram-бот, раздачу статики с фронтенда.

---

## Ключевые категории

### 1. Точки входа
- `backend/main.py` — запуск FastAPI
- `frontend/index.html` — запуск React-приложения
- `bot/bot.py` — запуск Telegram-бота

### 2. Конфигурационные файлы
- `docker-compose.yml`
- `backend/config.py`
- `frontend/vite.config.js`
- `frontend/nginx.conf`
- `nginx/default.conf`
- `.github/workflows/*.yml`
- `bot/bot.py` (частично)

### 3. Основные модули обработки данных
- `backend/ocr_service.py` — обработка изображений и извлечение данных через OCR
- `backend/mechanics.py` — реализация игровой механики
- `backend/database.py` — взаимодействие с базой данных
- `backend/models.py` — структура данных

---

> **Примечание**: Проект использует многоуровневую архитектуру с чётким разделением ответственности. Все сервисы независимы и могут развиваться отдельно, что упрощает масштабирование и тестирование.
